# LeaseFlow 관리자 경험 기준본

`EXPERIENCE.md`는 LeaseFlow 관리자 웹이 **어떻게 작동해야 하는지**를 정의한다. 시각 정체성과 토큰은 `DESIGN.md`를 따른다. 두 기준본은 화면 시안보다 우선하며, 기존 도메인·권한·게시·버전·감사 불변식을 바꾸지 않는다.

## Foundation

- **제품 표면:** Next.js 15 App Router 기반 반응형 관리자 웹. 모바일 운영 앱은 같은 도메인 모델과 감사 기록을 사용하지만 이 문서의 화면 범위는 관리자 웹이다.
- **사용자 역할:** 데이터 담당자는 원자료와 변경 후보를 확인한다. 선임 검토자는 승인·반려·게시를 결정한다. 임대 관리자는 게시된 운영 정보와 건물별 외부 보고를 사용한다.
- **업무 위험:** 반복적이며 외부 결과에 영향을 주는 고위험 운영 업무다. 정확성, 출처, 현재 버전, 역할 분리, 복구 가능성이 속도보다 우선한다.
- **시각 참조:** `DESIGN.md`의 `Quiet Evidence Ledger`와 `{colors.*}`, `{typography.*}`, `{components.*}` 토큰을 사용한다.
- **불변식:** 커뮤니케이션은 공식 정보를 덮어쓰지 않는다. GPT 결과는 후보·초안·패치다. 게시·활성·권한·외부 공유 조건을 모두 통과한 현재 정보만 외부 결과에 들어간다. 모든 외부 패키지와 보고서는 사람 승인이 필요하다.
- **데모 경계:** 역할 전환과 초기화는 업무 컨트롤이 아니라 별도 데모 도구다. 실제 Outlook, 전화, SSO, 회사 시스템 연동을 암시하지 않는다.

## Information Architecture

### IA 원칙

1. 전역 메뉴는 업무 도메인 사이를 이동한다.
2. 페이지 내 로컬 탐색은 같은 업무 도메인의 보기나 기록 범위를 바꾼다.
3. 전역 메뉴와 로컬 탭을 한 탐색 막대에 섞지 않는다.
4. 모든 탐색 항목은 실제 렌더링 가능한 route를 가져야 한다. route가 구현되지 않았으면 링크도 노출하지 않는다.
5. 앵커는 현재 페이지의 보조 점프에만 사용하며 전역 메뉴를 흉내 내지 않는다.
6. 필터, 검색, 정렬은 URL search params에 직렬화하여 뒤로 가기와 공유 가능한 맥락을 보존한다.

### 현재 구현과 G008 기준

G008 구현 전에는 `/`, `/reports`, `/mobile-preview`, `/design-showcase`만 페이지 route로 존재했고, 전역 메뉴가 `/`와 `/reports` 내부 앵커를 여러 업무 페이지처럼 표시했다. 이 진단은 구현 전 감사 기록이며 현재 상태를 설명하지 않는다.

현재 구현은 `/`, `/sources`, `/changes`, `/publishing`, `/operations`, `/reports`, `/settings`의 7개 전역 업무 route와 아래 로컬·상세 route를 제공한다. `/mobile-preview`는 데모 검증 표면, `/design-showcase`는 개발 참조 표면으로 남아 있으며 관리자 전역 메뉴에는 포함하지 않는다. 각 전역 메뉴는 실제 route에 연결되고 역할에 따라 허용 행동 또는 읽기·권한 안내를 표시한다.

아래 표는 **구현된 canonical route 계약**이다. 이후 route를 추가할 때도 페이지와 로딩·오류·권한 상태가 존재하기 전에는 메뉴를 노출하지 않는다.

G008 완료 당시 역사적 기준선은 Admin 77/77, 전체 146/146 tests, responsive 105/105였다. G009 이후 현재 최종 검증은 Admin 78/78, 전체 147/147 tests, 5/5 workspace typecheck, Admin production build 24 pages와 final 14-route viewport smoke를 통과했고 console error, HTTP failure, horizontal overflow는 모두 0이다. 최종 code review는 APPROVE, architect review는 CLEAR이며 권위 QA는 `artifacts/visual-qa/g008/G008_FINAL_QA.md`다.

### Canonical global route map

| 순서 | 전역 메뉴 | route | 기본 페이지 목적 | 주요 역할 |
| ---: | --- | --- | --- | --- |
| 1 | 오늘의 업무 | `/` | 역할별 처리 대기열과 다음 인계 확인 | 모든 관리자 역할 |
| 2 | 원자료 | `/sources` | 합성 원자료 검색·등록 범위 안내·상세 검토 | 데이터 담당자 |
| 3 | 변경 검토 | `/changes` | 현재 게시값과 후보값의 근거 비교·1차 확인 | 데이터 담당자 |
| 4 | 승인·게시 | `/publishing` | 선임 승인·반려·게시와 게시 결과 확인 | 선임 검토자 |
| 5 | 운영 정보 | `/operations` | 현재 게시된 사실·파일과 버전 이력 조회 | 승인된 관리자 |
| 6 | 임대인 보고 | `/reports` | 건물별 외부 보고 초안·패치·승인·발송 기록 | 임대 관리자 |
| 7 | 설정·기록 | `/settings` | 수신자 그룹·접근 권한·감사 기록 | 권한 있는 관리자 |

`대시보드`, `건물`, `보고그룹`, `보고일정`, `사용자·권한`, `발송·감사`를 각각 긴 전역 메뉴로 유지하지 않는다. 사용자의 실제 흐름에 따라 오늘의 업무, 운영 정보, 임대인 보고, 설정·기록 안으로 재배치한다. 도메인 책임은 삭제하지 않고 탐색 계층만 정리한다.

### Heading contract

- 페이지마다 `h1`은 정확히 하나다.
- 업무 구역마다 `h2`, 구역 안의 비교 항목·문서 섹션·기록 묶음마다 `h3`를 쓴다.
- `section[aria-labelledby]`의 참조 ID는 페이지에서 유일해야 한다. 같은 요소에 `id`와 heading ID를 중복 부여하지 않는다.
- route 전환 후 `h1`로 프로그램 포커스를 옮기고 `tabindex="-1"`을 사용한다. 이전 focus ring은 route 이동 직후 보인다.
- 고정 appbar 아래의 `h1`, `h2`, 오류 요약, skip target에 `scroll-margin-block-start`를 적용한다.

## Voice and Tone

업무 언어는 짧고 직접적이며 원인과 결과가 분명해야 한다. 브랜드 태도는 `DESIGN.md.Brand & Style`이 소유한다.

| 상황 | 사용 | 금지 |
| --- | --- | --- |
| 상태 | `검토 필요`, `승인 대기`, `현재 사용`, `이전 정보` | 내부 상태값 그대로 표시 |
| 오류 | `변경 사항을 저장하지 못했습니다. 입력은 유지했습니다. 다시 시도해 주세요.` | 서버 응답, 영문 예외, 상태 코드 |
| 권한 | `선임 검토자만 게시할 수 있습니다. 현재 내용은 읽을 수 있습니다.` | `Forbidden`, 역할 코드 |
| 충돌 | `다른 담당자가 먼저 변경했습니다. 최신 내용을 불러온 뒤 다시 확인해 주세요.` | revision 번호, optimistic concurrency |
| 성공 | `1차 확인을 마쳤습니다. 선임 검토 대기열로 이동했습니다.` | `성공`, 체크 아이콘만 |
| 비활성 | `원자료 확인이 끝나면 선임 검토로 인계됩니다.` | 미래 단계 비활성 버튼 |

설명은 사용자가 결정을 내리거나 실패에서 복구하는 데 필요한 경우에만 쓴다. 기술 정책을 상시 노출하지 않는다.

### 사용자용 한국어 label mapper 계약

1. API/도메인의 enum, event key, field key, role key, error code는 타입별 mapper를 거쳐야 한다.
2. mapper는 `Record<DomainType, KoreanLabel>`처럼 컴파일 시 누락을 드러내거나 exhaustive `switch`를 사용한다.
3. raw 값을 문자열 보간으로 fallback하지 않는다. 매핑 누락 시 화면에는 중립 한국어 `표시할 수 없는 상태` 또는 해당 문맥의 안전 라벨을 보여주고, 내부 관측 채널에만 누락 키를 기록한다.
4. 객체나 배열을 `JSON.stringify`로 사용자 화면에 출력하지 않는다. 각 도메인 항목을 `dl`, `ul`, `table`로 명시적으로 렌더링한다.
5. 내부 ID는 React key, API 요청, 로그 상관관계에만 사용한다. 사용자에게는 자료명, 건물명, 기간, 버전 번호처럼 의미 있는 참조를 보여준다.
6. confidence는 출처 수, 직접 확인 여부, 불일치 이유처럼 사용자 행동으로 이어지는 근거가 있을 때만 품질 라벨로 번역한다. 근거 없는 숫자 백분율은 숨긴다.
7. 날짜·시간·숫자·면적은 `Intl.DateTimeFormat`, `Intl.NumberFormat`의 `ko-KR` locale을 사용한다. 저장 문자열을 그대로 노출하지 않는다.

## Component Patterns

시각 사양은 `DESIGN.md.Components`와 `{components.*}`를 따른다.

| 구성요소 | 사용 위치 | 행동 계약 |
| --- | --- | --- |
| GlobalNav | 전 페이지 | 실제 route만 렌더링, `aria-current="page"`, desktop sidebar / smaller drawer |
| LocalNav | 운영 정보·설정·기록 | 전역 nav 아래, 실제 nested route, 현재 탭 `aria-current="page"` |
| TaskQueue | 오늘·검토·승인·보고 | 행 전체가 상세 링크, 필터 URL 보존, 선택 한 건만 활성 |
| SourceRegistry | 원자료 | desktop semantic table, mobile priority `dl` rows, detail 연결 |
| EvidenceViewer | 원자료·변경·승인 | 실제 파일 미리보기 또는 명시적 미지원 상태; 가짜 도면 금지 |
| ComparisonList | 변경·승인·보고 패치 | 항목별 현재/제안/근거, 키보드 순서가 시각 순서와 동일 |
| DecisionRail | 변경·승인·보고 | 현재 단계의 primary action 하나, 차단 원인과 인계 상태 |
| WorkflowHandoff | 모든 단계형 업무 | 미래 단계는 읽기 전용 텍스트, 담당 역할과 조건 표시 |
| StatusLabel | 목록·상세 | 사용자용 한국어 mapper, 색+텍스트, 내부값 없음 |
| FeedbackPanel | 로딩·빈 상태·오류·성공 | 원인·영향·복구, 해당 영역에 배치, live region 최소 사용 |
| DemoTools | 데모 전용 | 역할 전환·초기화·데모 경계; 업무 컨트롤과 별도 landmark |
| AuditTimeline | 설정·기록 | 시간순 `ol`, 사람이 읽는 사건·역할·현지화 시간 |
| VersionTable | 운영 정보 | 현재/이전/차단 상태, 현재 행 우선, 이전 정보 읽기 전용 |

### 단계별 한 행동

| 단계 | 주 행동 | 보조 행동 | 미래 상태 표현 |
| --- | --- | --- | --- |
| 자료 등록 안내 | `등록된 합성 원자료 보기` | 없음 | 실제 업로드 없이 원자료 상세에서 검토 시작 |
| 변경 추출 | `변경 내용 찾기` | 원자료 다시 선택 | `추출 후 데이터 담당자 확인` |
| 1차 검토 | `1차 확인 완료` | 값 수정 / 제외 / 추가 확인 요청 | `완료 후 선임 승인 대기` |
| 선임 검토 | `승인하고 게시` | 수정 요청 / 반려 | `게시 후 현장 공유 가능` |
| 보고 패치 | `변경안 반영` | 기존 내용 유지 | `반영 후 최종 승인 가능` |
| 보고 승인 | `보고서 승인` | 내용 수정 | `승인 후 발송 기록 가능` |
| 발송 기록 | `확인하고 발송 기록 남기기` | 이전 단계로 돌아가기 | 완료 상태만 표시 |

한 화면에 여러 단계가 보이더라도 현재 단계의 primary action만 버튼이다. 이전 단계는 완료 상태, 다음 단계는 인계 상태다.

## State Patterns

### 상태 매트릭스

| 상태 | 원인 | 사용자 영향 | 화면 처리 | 복구 행동 |
| --- | --- | --- | --- | --- |
| loading | 최초 조회 또는 route 전환 | 아직 판단 불가 | 실제 행·제목 구조와 같은 skeleton, 영역 `aria-busy` | 자동 완료; 10초 이상이면 다시 시도 제공 |
| empty | 조건에 맞는 업무·자료·기록 없음 | 현재 목록에서 할 일 없음 | 무엇이 비었는지와 범위를 명시 | 권한 있는 경우 한 개의 생성/이동 행동 |
| error | 조회·저장·검증 실패 | 결과 미반영 또는 화면 미완성 | 원인 범주, 보존된 입력, 영향 범위를 한국어로 표시; 여러 조회를 합성한 감사 표면은 하나라도 실패하면 모든 부분 기록을 숨김 | 같은 영역에서 `다시 시도`; 반복 실패 시 안전한 상위 route |
| retry | 일시 실패 후 사용자가 재요청 가능 | 작업 보류 | 기존 입력·선택 유지, 재시도 중 단일 busy | 동일 idempotency/expected state로 안전한 재시도 |
| success | 명령이 확정됨 | 다음 단계 또는 완료 | 결과와 인계 대상 알림, 관련 h2/h1로 focus 이동 | 다음 실제 route 링크 또는 없음 |
| permission | 역할·건물·그룹 권한 없음 | 변경/조회 일부 제한 | 숨길 정보는 숨기고, 허용 범위만 표시 | 요청 가능한 담당 역할 또는 이전 route 안내 |
| read-only | 이전 버전, 완료 기록, 권한 제한 | 수정 불가 | `읽기 전용` 이유와 시점 표시, 편집 컨트롤 제거 | 현재 버전 보기 또는 담당자 인계 |
| stale | 기준 정보·파일·수신자 변경 | 기존 초안 외부 사용 불가 | 무엇이 바뀌어 무효인지 표시, 승인/발송 차단 | 최신 게시 정보로 새 초안 만들기 |
| conflict | 다른 사용자가 먼저 저장 | 현재 편집을 바로 확정할 수 없음 | 사용자 입력을 보존하고 변경된 항목 요약 | 최신 내용 불러오기 → 비교 → 다시 제출 |
| disabled | 현재 단계 조건 미충족 또는 busy | 행동 불가 | 미래 행동은 버튼 제거, 현재 행동만 이유와 함께 비활성 가능 | 충족 조건 또는 진행 중 상태를 인접 텍스트로 표시 |

### 상태별 추가 계약

- `/settings/audit`의 보고 endpoint가 실패하면 화면은 fail closed로 전환한다. 다른 endpoint에서 받은 감사 항목도 노출하지 않고 안전한 오류 설명과 `다시 시도`만 표시한다. 재시도가 성공하면 완전한 감사 기록을 복원한다.

- 목록 loading과 상세 loading을 별도 영역으로 관리하되, 한 mutation에서 관련 제출 컨트롤은 모두 같은 busy 상태를 공유한다.
- 오류 후 입력·선택·필터·스크롤을 잃지 않는다.
- stale 상태에서는 이전 파일·평면도·면적·조건을 외부 패키지에 넣지 않는다.
- conflict 복구는 덮어쓰기 버튼을 제공하지 않는다. 최신 정보와 사용자 변경을 다시 비교한다.
- permission과 read-only는 같은 뜻이 아니다. 권한 제한은 역할 이유, 읽기 전용은 버전·완료 상태 이유를 설명한다.

## Interaction Primitives

- **포인터:** 행 전체 클릭은 상세 이동 하나만 수행한다. 행 안의 추가 메뉴가 필요하면 명확한 별도 버튼과 이름을 둔다.
- **키보드:** Tab 순서는 전역 탐색 → 로컬 탐색 → 페이지 제목 → 필터 → 대기열 → 근거 → 결정이다. `Esc`는 최상위 drawer/dialog만 닫고 입력을 폐기하지 않는다.
- **폼:** 오류 요약은 첫 오류 필드로 이동할 수 있는 링크 목록이다. 저장 실패 후 값은 유지한다.
- **route 전환:** 새 페이지의 h1에 focus. 뒤로 가기는 검색어, 필터, 정렬, 선택 행, 스크롤 위치를 복원한다.
- **dialog/drawer:** 한 번에 한 계층만 연다. 열 때 첫 의미 있는 제목 또는 컨트롤로 focus, 닫을 때 trigger로 focus 복귀.
- **파괴적 행동:** 데모 초기화 같은 파괴 행동은 데모 도구 영역에서 명시적 확인을 거친다. 일반 업무 주 행동과 같은 cluster에 두지 않는다.
- **busy:** 같은 업무에 중복 mutation을 허용하지 않는다. 로딩 텍스트와 `aria-busy`를 제공하며 버튼 폭은 유지한다.
- **금지:** hover-only 행동, drag-only 정렬, 무한 스크롤, 자동 진행, 자동 승인, 전역 nav 가로 스크롤, 중첩 modal.

## Responsive & Platform

| 범위 | 전역 탐색 | 업무 구성 | 표/목록 | 결정 레일 |
| --- | --- | --- | --- | --- |
| `≥1024px` | 232px sidebar 고정 | queue + evidence + rail 또는 evidence + rail | 전체 열, sticky header 허용 | 288–320px sticky |
| `768–1023px` | compact topbar + menu drawer | 한 열 또는 queue/evidence 2열 | 우선 열 유지, 상세 아래 배치 | 필요 시 우측 drawer, inline summary 유지 |
| `<768px` | appbar + menu drawer | 완전한 한 열 | 행을 의미 있는 `dl`로 변환 | 콘텐츠 뒤 inline, 주 행동 sticky 금지 또는 safe-area 고려 |
| `320/375px` | 아이콘+명확한 메뉴 이름 | 16px gutter, 44px targets | 핵심 2–3 정보만 기본 노출 | 한 행동 전체 폭 |
| 200% zoom | drawer 또는 재배치 | 한 방향 세로 흐름 | 우선 열 축소/재구성 | sticky 해제 가능 |

전역 메뉴를 가로 스크롤로 축소하지 않는다. 표가 2차원 스크롤을 요구하면 정보 우선순위를 다시 설계한다. 긴 파일명·한국어/영문 혼합·200% 확대에서 `overflow-wrap:anywhere`, `min-width:0`, 논리 속성을 사용한다.

## Accessibility Floor

- WCAG 2.2 AA를 최소 기준으로 한다.
- 모든 pointer target은 최소 44×44px다.
- `:focus-visible`은 모든 상호작용 요소에 보이며, light surface와 navy rail 모두 인접색 대비 3:1 이상이다. 현재 확인된 rail focus 후보는 8.12:1이다.
- 첫 focus 가능한 요소 전에 `본문으로 바로가기`를 제공한다. target은 페이지의 `main` 또는 h1이고 유일한 ID를 가진다.
- 고정 header로 가려질 수 있는 target과 heading에 `scroll-margin-block-start`를 적용한다.
- route 전환 후 h1 focus, dialog 닫힘 후 trigger focus 복귀, 오류 제출 후 오류 요약 focus를 보장한다.
- `nav`, `main`, `aside`, `section`, `table`, `dl`, `ul`, `ol`을 의미에 맞게 사용한다. 클릭 가능한 `div`를 만들지 않는다.
- 표의 열 헤더는 `scope="col"`, 모바일 재구성은 `dt/dd`, 기록은 `ol/time`을 쓴다.
- live region은 완료·오류·비동기 상태 변화에만 사용하고, 페이지 전체를 live region으로 만들지 않는다.
- `prefers-reduced-motion`에서 이동·순차 등장·회전을 제거한다.
- `forced-colors: active`에서 경계, focus, 선택, 오류가 시스템 색으로 남아야 한다. 색상 이미지를 강제로 유지하지 않는다.
- `Intl`의 `ko-KR` locale로 날짜·시간·숫자·단위를 형식화한다.
- 200% 확대에서 본문 2차원 스크롤, 텍스트 잘림, 겹침, 기능 손실이 없어야 한다.
- 320, 375, 768, 1024, 1280px과 키보드 전용, screen reader landmark 순서, long label을 검증한다.

## Inspiration & Anti-patterns

- **유지:** v3.0.0의 warm paper, ink rail, evidence/comparison, human decision 경계.
- **정제:** 기존의 큰 제목은 24–28px로 낮추고, 장식적 hero가 아닌 작업 제목으로 사용한다.
- **정제:** 기존의 고정 8개 IA는 책임을 보존하되 실제 사용자 흐름 기준 7개 업무 도메인으로 재구성한다.
- **거부:** 카드 벽, 가짜 KPI, CSS 가짜 평면도, 8943px 단일 작업 페이지, 내부 상태/ID/JSON 노출.
- **거부:** Dark OLED, Cinzel, Josefin, teal 자동 추천. 운영 대장의 고밀도 한국어 검토와 맞지 않는다.
- **채택:** 자동 추천 중 접근성·상호작용 체크리스트만 사용한다.

## Key Flows

### Flow 1 — 원자료에서 게시까지: 데이터 담당자 → 선임 검토자

1. 데이터 담당자가 `/`의 오늘의 업무에서 `원자료 확인 필요` 행을 연다.
2. `/sources/[sourceRef]`에서 파일명, 제공처, 기준일, 건물, 분류 제안과 실제 원문을 확인한다.
3. 지원되는 자료라면 `변경 내용 찾기` 한 행동으로 후보를 만든다. GPT 결과는 아직 공식 정보가 아니다.
4. `/changes/[batchRef]`에서 원문 근거, 현재 게시값, 제안값을 항목별로 비교한다.
5. 잘못된 값은 수정하고, 근거가 없으면 제외 또는 추가 확인 요청으로 남긴다.
6. **Climax:** `1차 확인 완료`를 실행하면 데이터 담당자의 편집 권한은 완료 상태가 되고 `선임 승인 대기` 인계가 표시된다.
7. 선임 검토자가 `/publishing`의 대기열에서 같은 묶음을 연다.
8. 원문, 현재값, 제안값, 적용일, 외부 공유 가능 여부를 다시 확인한다.
9. `승인하고 게시`를 실행한다. deterministic code가 역할, 현재 revision, 활성 버전, 파일 연결을 확인한다.
10. 게시 후 `/operations`에 새 현재 버전이 보이고 이전 버전은 읽기 전용이 된다. 새 평면도라면 기존 평면도는 외부 사용에서 즉시 차단된다.

실패: 저장 충돌이면 입력을 유지하고 최신 항목과 차이를 보여준다. 권한이 다르면 변경 버튼을 제거하고 필요한 담당 역할을 알려준다. 게시 실패면 공식 정보는 바뀌지 않으며 같은 결정 레일에서 안전한 재시도를 제공한다.

### Flow 2 — 건물별 임대인 보고: 임대 관리자

1. 임대 관리자가 `/reports`에서 이번 기간의 건물별 보고 대기열을 본다.
2. 보고 상세에서 현재 게시·활성·외부 공유 가능한 정보만 기준으로 초안을 만든다.
3. 저장된 업무 기록과 합성 이메일 자료를 조사해 source-backed patch 후보를 받는다.
4. 현재 보고 내용과 제안 변경을 항목별로 비교한다. 객체 덤프나 raw JSON은 보이지 않는다.
5. 근거가 충분하면 `변경안 반영`, 아니면 `기존 내용 유지`를 선택한다.
6. 설정된 수신자 그룹의 To/Cc, 건물명, 보고기간, 첨부 최신 버전을 확인한다. 모델이 수신자를 만들지 않는다.
7. **Climax:** `보고서 승인`으로 외부 결과를 고정한다. 승인 전에는 발송 기록 행동이 나타나지 않는다.
8. `확인하고 발송 기록 남기기`를 실행한다. 데모에서는 실제 이메일 대신 발송 기록만 생성한다.

실패: 기준 정보나 수신자 그룹이 바뀌면 stale 상태가 되어 승인·발송을 차단하고 `최신 정보로 새 초안 만들기`만 제공한다.

### Flow 3 — 이전 정보 확인과 현재 정보 복귀: 승인된 관리자

1. 관리자가 `/operations/versions`에서 특정 항목의 이력을 연다.
2. 이전 버전은 `읽기 전용 · 이전 정보`로 표시되고 사용 기간과 교체된 현재 버전이 보인다.
3. 평면도 또는 스태킹 플랜 이력에서는 현재 외부 사용 가능 파일을 먼저 보여준다.
4. **Climax:** `현재 정보 보기`로 `/operations`에 돌아오면 동일 건물·항목 필터와 포커스가 복원되고, 현재 버전이 선택되어 있다.

금지: 이전 파일을 외부 패키지에 추가하거나, 이력 화면에서 공식 데이터를 수정하는 행동.

## Verification Contract

| 검증 축 | 최소 시나리오 | 통과 조건 |
| --- | --- | --- |
| route | 전역 7 route + 모든 local route 직접 진입 | 404 없음, 실제 페이지 렌더, 올바른 `aria-current` |
| 계층 | 각 page의 heading outline | h1 하나, h2 업무, h3 항목, ID 중복 없음 |
| 행동 | 각 단계의 허용/미허용 상태 | primary action 하나, 미래 단계는 인계 텍스트 |
